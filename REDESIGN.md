## 一、原版游戏设计提炼（必须先吃透，才能决定保留 / 改）

### 1.1 玩法循环
1. **登录 / 注册**（UDP 广播找服务器 → SQLite 校验密码）
2. 进入**大地图**，方向键走动，走到不同建筑触发功能：
   - **竞技场 (Arena)**：对战练级
   - **道馆 (Gym)**：对战拿徽章 / 抓新精灵
   - **音乐 (Music)**：背景音乐开关
   - **出口 (Exit)**：下线
3. **对战准备**：左边选自己的一只精灵，右边从「虚拟精灵池」选一只敌人 → 本质是 **PvE**，不是玩家对玩家
4. **战斗**：实时自动战斗（双方按各自攻击间隔互相打），玩家只是观战
5. **结算**：赢了加经验、可能升级、可能抓到新精灵（最多 7 只）；输了精灵死亡可能被删

### 1.2 精灵数值模型（来自 `petcreating.cpp`，**这是游戏的灵魂，要原样搬过去**）

**基础属性（1 级）**：`atk=15, def=7, hp=300, interval=1.0s, exp=0`

**4 种类型 / 出生修正**（`changetype`）：

| 类型 | 中文 | atk | def | hp | interval | 特性技能 |
|------|------|-----|-----|----|----|--------|
| 1 str | 力量型 | +4 | −2 | — | — | 英勇打击：3 回合内攻击附加 `lvl×3` 真伤 |
| 2 fat | 肉盾型 | −1 | +1 | +20 | — | 生命汲取：回复 `atk×2` 血量 |
| 3 def | 防御型 | −2 | +3 | — | — | 石化表皮：3 回合内减伤 `lvl×3` |
| 4 agi | 敏捷型 | +1 | −1 | — | −0.1 | 眩晕：使对方下 2 回合无法攻击 |

**升级修正**（`lvlup`，每级，**满级 15**）：

| 类型 | atk | def | hp | interval |
|------|-----|-----|----|----|
| str | +12 | +5 | +62 | — |
| fat | +7 | +6 | +108 | — |
| def | +7 | +7 | +60 | — |
| agi | +8 | +5 | +50 | −0.03 |

**经验**：升级所需 `exp = 5×level`；击败对方获得 `gain = 10×对方level`。

**战斗判定**（每次攻击 `beat`）：
- 15% 概率放**技能**（`rans>85`），否则普攻
- 命中后计算：`damage = atk - 对方def`（最低 0）
  - str 技能加成：`+lvl×3`；def 技能减免：`−对方lvl×3`
- **闪避**（`damagedeal`，20% 触发 `dodge>80`）：agi 必闪；其他类型只有当伤害 ≤90 时才闪
- **暴击**（未闪避时 10% 触发 `crush>90`）：伤害 ×2
- 攻击节奏：基于 `clock()` 的 `interval` 秒间隔，**忙等待轮询**（待改）

> ⚠️ 数值里有几个明显的**原版 bug / 怪逻辑**，重写时要决定「忠实复刻」还是「顺手修」：
> 1. `dltPoke` 里 `index` 和 `number` 都读了 `at(0)`（[udpreceiver.cpp:167-168](server/udpreceiver.cpp#L167)）——删精灵逻辑很可疑
> 2. 闪避「伤害>90 不能闪」的规则很反直觉（越疼越闪不掉）
> 3. 随机数种子用 `qsrand`，且 str 技能的"真伤"实现是临时改 `state` 标记，耦合很重

### 1.3 12 只精灵
力量型：Hitmonlee、Charmander、Squirtle
肉盾型：Licktung、Muk、Krabby
防御型：Geodude、Shellder、Onix
敏捷型：Bulbasaur、Pidgeotto、Pikachu

### 1.4 数据模型（SQLite）
- `player`：一张**超宽表**，把 7 只精灵的 `pokemonN/levelN/expN` 全内联进列里（反范式，难维护）
- `pokemon`：虚拟精灵池（id, name, level），PvE 的敌人从这里选

## 二、新版技术选型（已确认：Web 版）

| 层 | 选型 | 理由 |
|----|------|------|
| 语言 | **TypeScript**（全栈共用类型） | 前后端共享精灵数值/协议类型，重构安全 |
| 前端 | **React + Vite** + Canvas（对战动画用 **PixiJS** 或纯 Canvas2D） | 随开随玩、HMR 反馈快 |
| 状态 | Zustand（轻量） | 比 Redux 省事，够用 |
| 后端 | **Node + ws (WebSocket)** | 替代 UDP 广播，连接可靠、天然支持真 PvP |
| 数据库 | **SQLite (better-sqlite3)** 起步 | 单文件、零运维，后续可换 Postgres |
| 协议 | **JSON over WebSocket**，带 `type` 字段 + zod 校验 | 替代裸 QDataStream + 魔法数字操作码 |
| 鉴权 | 注册时 **bcrypt** 哈希密码 | 修掉原版明文存密码 |
| 测试 | Vitest（战斗引擎是纯函数，最该测） | 数值逻辑可回归 |

### 2.1 目标目录结构
```
Battle_Pokemon/
├─ legacy/                 # 把原 client/ server/ 移进来存档，只读参考
├─ packages/
│  ├─ shared/             # 前后端共享：精灵数值表、类型、协议、战斗引擎（纯函数）
│  │  ├─ pokemon.ts       #   12 只精灵 + 4 类型数值（1.2 节那两张表）
│  │  ├─ battle.ts        #   战斗引擎，纯函数，可单测
│  │  └─ protocol.ts      #   WebSocket 消息类型 + zod schema
│  ├─ server/             # Node + ws + SQLite
│  └─ client/             # React + Vite + Canvas
└─ REDESIGN.md
```

### 2.2 核心架构决策
1. **战斗引擎做成纯函数 / 确定性状态机**：输入「双方精灵 + 随机种子」，输出「逐回合事件流 + 最终结果」。
   - 好处：可单测、可在前端预演也可服务端权威计算、彻底干掉忙等待
   - 前端只负责把事件流**回放成动画**（HP 条、飘字、技能特效）
2. **协议升级**：从 11 个魔法数字 → 命名的消息类型（`login`/`register`/`battleStart`/`battleEvent`/…），zod 运行时校验
3. **保留 PvE，预留 PvP**：先 1:1 复刻"打虚拟精灵池"，但 WebSocket 架构让"打真人"几乎零成本加上去

## 三、已确认的设计决策（2026-06-07 拍板）

1. **数值忠实度**：以 1.2 节数值为**基线复刻**，但**顺手调平衡** ——
   明显失衡处会修正（例：fat 型升级 +108 hp 远超其他类型，str 型 +62 hp+12 atk 偏暴力）。
   原则：每个类型有清晰定位、不出现碾压级强势/废柴类型。具体调整值在 `shared/pokemon.ts` 里加注释标明「原版→新版」。
2. **原版怪 bug：全部修掉** —— 删精灵的 `index/number` 重复读取、"伤害>90 不能闪"的反直觉闪避规则，
   都改成合理实现。技能"真伤"用干净的伤害管线表达，不再靠临时改 `state` 标记。
3. **PvP：这次就做** —— WebSocket 架构下加真人对战。战斗引擎服务端权威计算，两端回放同一事件流。
   PvE（打虚拟精灵池）与 PvP（打在线真人）共用同一战斗引擎。
4. **美术：先占位图 / 纯 Canvas 几何图形** 跑通玩法，贴图后补。每个精灵给一个类型色 + 名字标签即可。
5. **legacy 处理**：旧 `client/`、`server/` 移进 `legacy/` 存档（git mv 保留历史）。
