# 美术资源方案：多状态图 / 动图（Italian Brainrot）

> 目标（用户已定）：**多状态图 / 动图**路线——每个角色有待机/受击/攻击/死亡等表现，
> 越好看越好。项目去向为**自娱 / Demo / 学习**，现阶段不把版权当硬约束（可用抓来的 meme 图）。
> 本文是**决策方案**，不含实现代码。

---

## 0. 现状（已核对代码）

- 战场是 **Canvas 程序化绘制**（[BattleStage.tsx](packages/client/src/BattleStage.tsx)）：
  每个角色 = 按属性配色的圆 + 朝向小三角 + emoji 状态徽章（☠/⬇/💫）+ HP 条。
- **零外部美术资产**：无 `public/`、无图片、Vite 默认配置。
- 角色→视觉映射在 [presentation.ts](packages/client/src/presentation.ts)（目前仅"最高属性→主题色"）。
- **关键架构事实**：[useBattle.ts](packages/client/src/useBattle.ts) 的 `step()` 把 `applyAction`
  吐出的整批事件**瞬时套用**——只把事件转成日志文本，然后一步 `setState` 跳到终态。
  **当前没有"逐事件时间轴回放"**，Canvas 直接重绘最终画面。

> ✅ 引擎已输出完整语义事件流（`hit`/`damage`/`lifesteal`/`heal`/`thorns`/`buff`/`downed`/`dead`/`end`），
> 美术/动画层是纯**渲染替换**，**不碰 shared 逻辑**、不破坏确定性与单测。

---

## 1. 三个 concern 的定性

| Concern | 本质 | 结论 |
|---|---|---|
| 1. 不一定有 SVG / 不适合放进战场 | brainrot 是 AI 位图，**无法矢量化** | 放弃 SVG 念头；走位图（PNG/WebP）即可，Canvas/`<img>` 都吃位图 |
| 2. 分辨率不清晰 / 图源杂 | 尺寸不一、带背景/水印 | 需要一道**统一预处理**（裁切+去背+统一尺寸）→ 资产管线解决 |
| 3. 需要受击/攻击/死亡多状态（动图） | 12 角色 × 多状态 = 几十份一致风格资产 | **真正的工程量**；下文用"程序化动画 + 单图"把它降维 |

**核心降维思路**：把"多状态表现"拆成两层——
- **状态表现（动画）= 代码做**：受击抖动+闪红、攻击突进、死亡淡出旋转、暴击放大……
  这些用 Canvas 变换（位移/缩放/透明度/色调）**对任意单张图都通用**，不需要逐状态画图。
- **角色身份 = 一张图**：每个角色只需**一张处理好的待机图**，动画由代码施加在它身上。

这样把 concern 3 从"几十张动画资产"压成"**12 张静态图 + 一套通用动画代码**"，
既拿到"动起来、有受击/攻击/死亡反馈"的效果，又把美术量控制在可行范围。

## 2. 必做的地基（独立于"图从哪来"）：事件时间轴回放

无论用不用真图，**要让战斗"动起来"，先得把瞬时结算改成逐事件回放**。这是所有动画的前提。

**改造点（仅 client，不动 shared）**：
- 新增一个**事件队列播放器**（`useBattlePlayback` 或扩展 [useBattle.ts](packages/client/src/useBattle.ts)）：
  `applyAction` 返回的 `events` 不再瞬时套用，而是**入队**，按节拍（如每 200–400ms）逐个 `dispatch`。
- 维护一个**渲染态**（`viewState`）：随事件推进逐步更新（hit→闪现命中数字、damage→扣血条+受击动画、
  downed→倒地动画、dead→消失），与"逻辑终态 `state`"分离。
- BattleStage 接收**每帧的视觉状态 + 当前活跃动画**，用 `requestAnimationFrame` 驱动补间。
- AI 自动推进（现有 `AI_DELAY_MS` 节拍）改为"等本次事件回放完再推进下一步"。

> 产出：即便仍用现在的圆形占位，战斗也会有"出手→命中跳字→受击抖动→倒地"的节奏感。
> 这层做完，换不换真图都受益。

## 3. 资产管线（解决 concern 1 & 2）

把"杂乱 meme 图"变成"战场能用的统一资产"，一次性脚本化：

1. **收集**：每角色找 1 张正面、主体清晰的图（self/Demo 用途，先不纠结版权）。放 `art/raw/<id>.*`。
2. **预处理脚本**（Node + `sharp`，新增 `art/process.mjs`）：
   - 去背景（优先找已去背的 PNG；否则用 `rembg` 之类工具/在线去背一次性处理）
   - 居中裁切到统一画布、统一输出尺寸（如 256×256）
   - 导出 **WebP**（体积小、带透明通道）到 `packages/client/public/fighters/<id>.webp`
3. **接入**：[presentation.ts](packages/client/src/presentation.ts) 增加 `fighterSprite(archetypeId): string`
   返回 `/fighters/<id>.webp`；BattleStage 用 `Image` 预加载 + `drawImage` 替换画圆。
   **找不到图的角色自动回退到现有圆形占位**（渐进式，不阻塞）。

> Vite 默认把 `public/` 原样发布，`/fighters/x.webp` 直接可用，无需额外配置。
> 资产是"可选增强"：缺图回退占位，12 张可以一张一张加。

## 4. 状态动画（解决 concern 3，全部代码实现，对单图通用）

| 引擎事件 | 视觉表现（Canvas 变换施加在角色图上） |
|---|---|
| `action`（出手） | 角色朝目标**突进**一下再回弹（位移补间） |
| `hit` 命中 / `crit` | 命中时目标**闪白/震屏**；暴击额外**放大**一下 + 跳更大的伤害数字 |
| `damage` | 目标**受击抖动**（左右小幅 shake）+ **闪红**；HP 条平滑下降 |
| `miss` | 目标轻微**侧移**（闪避感）+ 灰色 "MISS" |
| `lifesteal` / `heal` | 绿色 "+N" 上浮 + 角色短暂**泛绿** |
| `thorns` | 攻击者身上弹一下小尖刺图标 |
| `buff` | 角色脚下/头顶冒一个**增益光圈**（用现有 buff note 文本飘字） |
| `downed` | **倒地**：旋转 90° + 变半透明 + 灰度（已有半透明，加旋转/下沉补间） |
| `dead` | **淡出 + 缩小 + 旋转**消失，留一个 ☠ |

这些是纯 Canvas/RAF 动画，**与具体是圆还是真图无关**——先在圆形占位上做出来，
之后图一换，动画自动套用到图上。**这是把"动图"需求转成"代码动画 + 静态图"的关键。**

可选进阶（后续）：少数高光时刻（暴击/必杀技）做专门特效；个别角色若拿得到序列帧再单独升级。

## 5. 推荐落地顺序（渐进、每步可玩）

1. ✅ **Step A — 事件回放地基**（§2）：瞬时结算 → 逐事件时间轴（可调速 1x/2x/瞬间）。已落地。
2. ⬜ **Step B — 通用状态动画**（§4）：受击抖动/闪红、攻击突进、倒地/死亡补间、跳字。**下一步。**
3. ✅ **Step C — 资产管线 + 接图**（§3）：`art/process.mjs`（`npm run art`）+ BattleStage `drawImage`
   替换圆形占位、缺图回退、按队伍朝向翻转。已落地。
4. ✅ **Step D — 12 张真图接入**（含原生朝向表）。已落地。
5. ⬜ （可选）**Step E — 高光特效 / 局部序列帧**。

> A、B 与资产完全解耦、纯收益，建议无论如何先做。C 是"真图"的试金石——
> 一个角色跑通了，铺量只是重复劳动。

## 6. 取舍与风险

- **WebP 动图（animated WebP）/ GIF**：技术上 Canvas `drawImage` 不会自动播放动图帧；
  要逐帧得拆帧或用 `<img>` 叠层。**建议先不做真动图**，用§4 的代码动画 + 静态图达到"会动"的观感，
  性价比最高。真序列帧留到 Step E 个别角色。
- **风格统一**：抓来的 12 张 meme 图画风/光照不一，拼在一张战场会"脏"。预处理统一描边/底色光圈
  能缓解（每个角色一个圆形底盘 + 主题色光环，图嵌在盘里）。
- **版权**：当前自娱可忽略；若日后转公开发布，§3 的 `art/raw` 是替换点——换成原创/授权图即可，
  代码不动。
- **不影响引擎**：以上全部在 `packages/client`，shared 的确定性/单测不受影响。
